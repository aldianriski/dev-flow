## 1. P0 — Architecture Risk (Block Release / Adoption)

### P0-1 — `read-guard.js` is a no-op; Thin-Coordinator Rule unenforced

**Evidence**
- `scripts/read-guard.js:18` — `const COMPACT_VULNERABLE = new Set();` (empty)
- `scripts/read-guard.js:11` — comment: *"v2: no phase file is written — guard is always pass-through."*
- `scripts/read-guard.js:60` — `if (!existsSync(PHASE_FILE)) process.exit(0);` — the file is never created
- `docs/v2-rewrite-plan.md:78` — "All phase-tracking machinery" listed as **Deleted** in v2
- `docs/DECISIONS.md:40-49` (ADR-003) — still `Status: decided`, mandates `set-phase.js` and 6 compact-vulnerable phases. `set-phase.js` does not exist.

**Impact**
- The PreToolUse hook (`Read|Grep|Glob`) fires on every tool call, runs Node, exits 0. Pure overhead.
- Documented safety property "Thin Coordinator Rule" claims orchestrator never reads source files in compact-vulnerable phases. Currently false. Orchestrator can read anything, anytime.
- Adopters reading ARCHITECTURE.md §Security Boundaries trust a guard that does not guard.

**Recommendation**
Pick one path explicitly:
- **(a) Remove guard entirely** → delete `read-guard.js`, remove PreToolUse matcher from `plugin.json` + `hooks/hooks.json`, write ADR-013 superseding ADR-003, drop §Security Boundaries claim from ARCHITECTURE.md.
- **(b) Restore the contract** → reintroduce a phase-marker mechanism (could be lighter than v1's `set-phase.js` — e.g. orchestrator writes `.phase` directly via Write tool at phase boundaries), populate `COMPACT_VULNERABLE`, retest.

Currently the worst of both worlds: cost of running the hook with none of the protection.

### P0-2 — Hook config duplicated in two files

**Evidence**
- `.claude-plugin/plugin.json:9–43` — full `hooks` block (SessionStart `session-start.js`, PreToolUse `read-guard.js`, PreToolUse chain-guard).
- `hooks/hooks.json:1–38` — byte-identical block.

**Impact**
- Two sources of truth → one will drift after the next edit.
- Per Claude Code plugin spec, `hooks/hooks.json` is the canonical hook manifest. `hooks` key inside `plugin.json` is generally not a recognized field for that spec. If both are honored, every hook fires twice (verify with a smoke run). If only one is honored, the other is dead config.

**Recommendation**
- Delete the `hooks` key from `.claude-plugin/plugin.json` (keep only name/description/author/license/version).
- Keep `hooks/hooks.json` as canonical.
- Add a sentence to `CLAUDE.md` Anti-Patterns: "❌ Hooks defined in plugin.json — plugin spec uses `hooks/hooks.json`."

### P0-3 — Stale ADRs not marked superseded; adr-writer rule self-violated

**Evidence**
- ADR-003 (`docs/DECISIONS.md:40`) — `Status: decided`. References `set-phase.js`, 6 compact-vulnerable phases. Both gone in v2.
- ADR-007 (`docs/DECISIONS.md:104`) — `Status: decided`. Says: *"Add `mvp` mode with 3 phases: Parse → Implement → Close. Gate 0 and Gate 1 are skipped."* Current `skills/dev-flow/SKILL.md:47-54` mvp = **7 phases with G1+G2 required**.
- ADR-008 (`docs/DECISIONS.md:125`) — references "10 phases executed" in dogfood. Current model has fewer named phases.

- `skills/adr-writer/SKILL.md:69` — Hard Rule: *"Status must be exactly one of: `proposed`, `decided`, `deprecated`, `superseded-by ADR-NNN`."*

**Impact**
- A new contributor reading `DECISIONS.md` top-to-bottom believes the v1 mode model is current. They then read `skills/dev-flow/SKILL.md` and find a contradicting model. **WHY history is unreliable.**
- The skill's own hard rule is silently ignored — credibility cost on the entire ADR practice.

**Recommendation**
- Add ADR-013 ("v2 supersedes v1 phase/mode model") that explicitly supersedes ADR-003, ADR-007, ADR-008, ADR-009.
- Update each affected ADR's status line to `superseded-by ADR-013`.
- Ratify with a `/adr-writer` invocation (dogfood the rule).

### P0-6 — Phase-numbering contract mismatch (security-analyst Phase 7 doesn't exist)

**Evidence**
